## Context

The admin store review CRUD routes (index, update, delete, bulk delete, accept, reject) are fully implemented using Sylius resource routing and a programmatic grid. There are no functional tests covering these routes. The project already has a functional test pattern established in `ReviewControllerTest` for shop-side routes.

## Goals / Non-Goals

**Goals:**
- Functional test coverage for all admin store review CRUD routes
- Verify authentication is enforced on admin routes
- Verify store reply feature works through the update form
- Follow established project test patterns (WebTestCase, DAMA, fixture data)

**Non-Goals:**
- Testing grid filtering/sorting (Sylius framework concern)
- Testing form validation edge cases
- Testing workflow transition edge cases (e.g., double-accept)
- Unit testing (these are all functional/HTTP-level tests)

## Decisions

### Authentication: Use Symfony's `loginUser()` helper
**Rationale:** Symfony's `$client->loginUser($user, 'admin')` is the standard approach for functional tests. It avoids HTTP overhead and fragility of testing the login form itself. The admin user is fetched from Sylius default fixtures.

**Alternative considered:** HTTP login flow (GET login page → POST credentials). Rejected because it's verbose, slower, and tests the login system rather than our CRUD.

### Test data: Create StoreReview entities in each test
**Rationale:** Sylius fixtures don't include store reviews. Tests that need a review will create one via the entity manager using fixture Channel and Customer data. DAMA handles rollback.

### File location: `tests/Functional/Controller/Admin/StoreReviewControllerTest.php`
**Rationale:** Follows the existing pattern (`tests/Functional/Controller/ReviewControllerTest.php`) with an `Admin/` subdirectory to separate admin tests from shop tests.

### CSRF tokens: Use Symfony's CsrfTokenManager
**Rationale:** Delete, bulk delete, accept, and reject actions require CSRF tokens. Use `self::getContainer()->get('security.csrf.token_manager')` to generate valid tokens rather than scraping them from HTML.

## Risks / Trade-offs

- **[Fixture dependency]** → Tests depend on Sylius default fixtures having at least one Channel and one Customer. This is reliable since other tests already depend on the same fixtures.
- **[Sylius internal routes]** → Route paths are generated by Sylius resource routing and could change between Sylius versions. Mitigated by using route names via the router service where practical, and by hardcoding known paths that match the debug:router output.
